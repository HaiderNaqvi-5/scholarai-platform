00 — How v3 differs from v2

The codebase moved. The plan now matches it.

The v2 plan (FRONTEND_ENHANCEMENT_PLAN.html) was written before the Pakistan API layer landed. Since then, the backend ships and the frontend has wired: client.ts with a 402 PlanRequiredDetail guard, all 17 endpoint modules including tracker, scholarshipMatch, sopBuilder, visaInterview, upgrade and reports, the index.ts barrel, the UpgradeWall component, and a 6-card B2B profile page. The design tokens in globals.css already match the spec palette.

What v2 listed as "build the API module" or "new UpgradeWall component" is DONE. What remains is the student-facing pages: the Pakistan landing, the 5-step onboarding, the dashboard redesign, and the four feature pages (Scholarship Finder, Tracker, SOP Builder, Visa Interview), plus the Upgrade page and the privacy/consent surface. v3 re-scopes every sprint to that real gap and drops ~5 dev-days of already-finished work.

Brand decision baked into this plan

Public display brand is AidwiseAI everywhere; frontend/CLAUDE.md is updated to match. ScholarAI stays internal/repo only. The codebase currently hardcodes a previous brand string in app/page.tsx and Sidebar.tsx — rename it to AidwiseAI in S1/S3. The old DESIGN_SPEC.md instruction to rename the brand to "ScholarAI Pakistan" is not actioned.

03 — Design Principles for This Build

Non-negotiable rules — PR-blocking

These carry forward the nine UX rules from v2 and bind them to the hard constraints already in frontend/CLAUDE.md. They override default behaviour. GoScholar wins on emotion; AidwiseAI wins on depth, trust, and Pakistan specificity.

Trust

Validated is not AI-generated

Scholarship facts (deadlines, amounts, eligibility) get a validated-stripe + soft-green tint + "Verified" badge + source link. AI output (SOP draft, match reasons, feedback) gets a generated-stripe + soft-blue tint + one 14px Sparkles icon + one-line provenance. Never mixed in a paragraph.

Honesty

Show locked, never hide

Free tier sees blurred results with real counts. UpgradeWall renders the API 402 detail.message verbatim. Never write a generic "unlock this feature".

Speed

Zero AI theatre

No fake streaming, no typewriter on REST data, no spinner-on-every-card. Recommendations render once. Skeleton shows once per route load. SOP generation shows a milestone progress bar, not a spinner.

Focus

One primary CTA per screen

Landing to "Find My Scholarships". Onboarding to "Continue". Finder card to "Add to Tracker". No decision paralysis, no twin hero buttons.

Identity

Pakistan context everywhere

PKR pricing, Pakistani university names, CGPA/HEC context, visa-fear language. Every screen should feel built for a student in Lahore or Karachi.

Reach

Mobile-first, WhatsApp-aware

375px is the primary canvas. Bottom tab bar on mobile. Share-to-WhatsApp on scholarship cards. Every primary action is a 44x44 tap target.

Craft

No emoji as UI, no AI tells

Lucide SVG icons only; emoji allowed only in content (country flags from data). No purple glow, no gradient text, no model names, no "Powered by", no floating chat bubble, no "Ask AI" buttons. No em dashes in product copy.

Performance

Latency and bundle budgets

LCP < 1.8s on /feed (3G simulated). INP < 200ms on save toggle, kanban drag, filter change. Initial JS ≤ 180KB gzipped per route. Tree-shake Radix, no full-barrel imports. Animate only transform and opacity.

Resilience

Every state, every time

Loading (skeleton matching layout), empty (composed, with a CTA), error ("Couldn't load. [Retry]", never "Network error"). Optimistic mutations with rollback toast on 4xx.

Access

Keyboard and motion safe

/ focuses search anywhere, Esc closes any drawer/dialog. Visible focus rings (--color-ring). prefers-reduced-motion respected globally — all slide/fade degrade to instant, kanban drag still works.

04 — Unique Differentiators vs GoScholar

What AidwiseAI has that they don't, and must make visible

Visa Interview Simulator, first in market

GoScholar lists visa prep as "Coming Soon". AidwiseAI has 70 seeded questions for UK, USA, Canada, Germany with AI rubric evaluation, and the visaInterview.ts client is already wired. This is the headline feature: front-and-centre on landing and the dashboard action grid.

Pakistani CGPA conversion context

"Your 3.7 CGPA from NUST is approximately a 3.7 US GPA, First Class Honours (UK)." No competitor shows this. Surface it live in onboarding Step 2 and on every match card.

SOP with Pakistani academic context

Understands HEC degrees, Pakistani university prestige, ties-to-Pakistan framing for government scholarships, the visa-officer reading lens. GoScholar's assistant is generic.

HEC attestation awareness

Every tracker card's checklist has "HEC Degree Attestation, required for UK/Germany" as a highlighted row with an info tooltip. A Pakistan-specific pain point no other platform surfaces.

PKR-first freemium pricing

"Less than one consultant meeting, PKR 2,499/month." Positioned against the Pakistani consultant market, with a currency switcher for diaspora families in GBP/EUR/AED.

Multi-channel alerts (Elite)

Email + SMS + WhatsApp deadline alerts. WhatsApp is the communication layer in Pakistan; GoScholar has none. A real retention moat, surfaced on the Upgrade comparison table.

05 — Onboarding Flow Redesign

Pakistan-specific 5-step onboarding, under 3 minutes

Rewrite the current 4-step generic form. Keep the proven mechanics: the grantpath.onboarding_draft localStorage autosave (debounced), the Field / Chip / ProgressDots helpers, the endpoints.profile.upsert submit. Extend the step count to 5 and the field set to Pakistan-specific. Every field must map to a real StudentProfile column (extra="forbid") — confirm against types.ts first.

Step 1

About You

  • Full name (to User)
  • City (Karachi, Lahore, Islamabad...)
  • Pakistani university autocomplete (NUST, LUMS, FAST, UET, IBA, QAU...)
  • Degree subject
  • Graduation year
~45 seconds
Step 2

Academic Record

  • CGPA value + scale
  • Live US-GPA conversion (validated-stripe)
  • HEC degree level
  • Research publications toggle + count
~45 seconds
Step 3

Test Scores

  • IELTS / TOEFL (optional)
  • GRE Quant / Verbal (optional)
  • GRE waiver toggle
  • Clear "all optional" messaging
~30 seconds
Step 4

Your Goal

  • Target degree (MS/PhD/MBA)
  • Target countries (flag cards, multi)
  • Target fields (chips)
  • Funding requirement
  • Intake target
~45 seconds
Step 5

Financial Context

  • Afford application fees?
  • Family funds for bank statement?
  • GRE fee waiver needed?
  • Skip option visible
  • CTA to /feed
~20 seconds

Critical UX decisions for onboarding

1. Progress bar + step dots + "~3 min left" estimate, persistent across steps.
2. CGPA US-equivalent shows inline as the student types (debounced 300ms), in a validated-stripe — builds trust immediately.
3. Pakistani university field is a searchable combobox, 25 universities pre-loaded, free-text fallback.
4. Step 4 country select uses large tappable flag cards, not a dropdown — mobile-friendly and satisfying.
5. "Continue" is always enabled except Step 4 (needs target_degree + at least one country).
6. target_countries[] is multi-select; mirror the first choice into the legacy single target_country_code column.
7. Final step CTA reads "Find My Scholarships", not "Save".

10 — Design Methodology

How the design skills shaped this plan

The design system and UX principles above were pressure-tested against three design skills: impeccable, design-taste-frontend, and ui-ux-pro-max. Where a skill conflicted with a project rule already codified in frontend/CLAUDE.md, the project rule won — that file is authoritative.

impeccable

Color strategy + anti-slop

Palette declared as Restrained: warm tinted neutrals plus semantic accents used only for meaning, one gold identity accent under ~10%. No #000/#fff, no gradient text, no glassmorphism-by-default, no hero-metric template, no identical card grids. Hero is asymmetric, not centered. No em dashes in product copy.

impeccable — exception

The stripe carve-out

impeccable bans >1px side-stripe borders. AidwiseAI keeps validated/generated/caution-stripe because frontend/CLAUDE.md rule 1 makes the validated-vs-AI partition PR-blocking. Paired with a full border + tint + badge + icon, the stripe is a semantic affordance, not decoration — a documented, intentional exception.

design-taste-frontend

Component architecture + bias correction

Interactivity isolated in 'use client' leaf components; static layout in Server Components. Cards used only where elevation communicates hierarchy, never nested, grouping otherwise via border-t/divide-y/space. Mandatory loading + empty + error states. Animate only transform/opacity. No "Jane Doe" placeholder data, no startup-slop names.

ui-ux-pro-max

Metric-based priority rules

Accessibility and touch targets are CRITICAL: 4.5:1 contrast, visible focus rings, aria-labels on icon-only buttons, 44x44 minimum (the tap-target utility already enforces this). Performance HIGH: reserve space for async content, respect prefers-reduced-motion, 150–300ms micro-interactions. SVG icons only, never emoji as UI.
